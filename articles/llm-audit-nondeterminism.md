# 同じコードをローカルLLMに51回監査させたら：多数決は"正しさ"より"しつこさ"を選んでいた

> この記事が答えること：**同じコードを同じLLMに何度もレビューさせると、指摘はどれだけ一致するのか**。そして「1回では不安だから複数回まわして多数決を取ろう」という直感が、どこまで本当かを実測で確かめる。
> 対象：`Qwen2.5-Coder`（1.5B / 7B, Apache-2.0）を `Ollama` でローカル実行。環境：NVIDIA RTX 4060 Ti（CUDA）、量子化 Q4_K_M、Ollama 0.31.1。全コード・全ログ・グラウンドトゥルースを公開。
> 誠実さの宣言：**自分の思い込み・そして"測定器そのもののバグ"まで含めて訂正過程を全部出す**。この記事のいちばんの見どころは §8 の落とし穴ログです。

## 30秒でわかる結論

温度を 0 にして、実行環境（GPU・量子化）まで固定すれば、**本実験の固定条件では、ローカルのAIコードレビューは全反復で同じ指摘を返しました**（乱数の種は、温度0では効かないことが分かりました——後述）。「AIの出力は毎回バラつく」という雑な理解は、少なくとも"完全に固定した条件"では当たりません。

ところが、実務でよくある「少し多様性を出す設定」にした瞬間、**同じコードなのに指摘がバラつきます**。1回のレビューをそのまま信じて直すのは危うい。

では「何回もレビューさせて、みんなが言う指摘だけ採用（多数決）」すれば安全でしょうか。**いいえ。** 多数決が残すのは「**しつこく出てくる指摘**」であって、「**正しい指摘**」とは限りませんでした。しつこいパターン反応は生き残り、たまにしか気づかない本物のバグは（指す場所がブレて）消える。安全網のつもりが、そうはならない。

*この下は、それを凍結ログと集計で実際に確かめていく記事です。技術的な詳細に興味がなければ、ここまでで要点は伝わります（"多数決が安全網にならない理由"は §7、"測定器のバグを自分で見つけた話"は §8 に）。*

> ⚠ 再現性の射程：**分散の"存在"と"桁感"は再現可能**ですが、厳密な数値は環境依存（GPU/量子化/推論エンジンの版）です。本記事は生成ログを凍結同梱し、読者は**決定的な集計だけ**を再現します（§10）。

## TL;DR

> 観察対象＝Qwen2.5-Coder 2サイズ・欠陥を仕込んだ小コード群・N=51回反復（温度0基準はN=11）。一般法則ではなく、**この標本・このGPU・この量子化での観察**。

1. **固定すれば決定的**：temp0・seed固定・GPU固定で、指摘集合の実行間一致（**Jaccard**：2回の指摘が何割重なるか。1.0＝完全一致）は **1.00**（4条件すべて、実行間で変動なし）。seed を6通り変えても出力はバイト単位で一致＝**temp0ではseedは効かない**。
2. **温度を上げると崩れる**：temp0.7・欠陥入り `target_a` で Jaccard は 7B **0.52**、1.5B **0.07**。7Bは0.52が**行ズレを1〜2行まで許すと 0.80** まで戻る＝不一致の大半は「**同じバグを違う行に指す**」ドリフト。（この2サイズ比較では 1.5B の方が大きく崩れた。）
3. **核心＝多数決は"しつこさ"を拾う**：一貫して出る指摘は多数決を生き残る。逆に、本物のバグでも**指す行がブレて票が割れる**と、少し厳しめの閾値では**どの行も過半に届かず落ちる**。＝**多数決が見ているのは"パターンの顕著さ"で、"正しさ"ではない**。
4. **正直な注意**（すべて `target_a`・per-run）：カテゴリ厳格な recall は低い（7B 0.31、1.5B 0.05）。ただし 7B は precision 0.98＝**出せば当たる**。1.5B は51回でファイル32行中25行を叩く"多弁"かつ低recallだったため、主張の主軸にはせず対照群として扱い、主張は7Bで立てる。

## 1. なぜこの検証をやるのか（動機と白地）

> 要点：AIを"レビュアー"として使うとき、その出力が**実行ごとにどれだけ揺れるか**を、ローカル・無料・凍結ログ付きで測った例が見当たらなかった。

LLMを評価者・監査者に使う "LLM-as-a-Judge" は普及していますが、出力の**非決定性**という弱点があります。先行研究はスコアの安定性を測っていますが（[Klishevich ら arXiv:2502.20747](https://arxiv.org/abs/2502.20747) は 70 コミット×温度0×5反復・**相関係数によるスカラ判定の一貫性**、[Insight Edge](https://techblog.insightedge.jp/entry/llm-as-a-judge-determinism) はクラウドの"同一スコア出現率"）、スコアではなく**"指摘の集合そのもの"が実行ごとに揺れるか**を、ローカル・無料・生成ログ凍結で扱った例は、**少なくとも筆者が確認した範囲では**見当たりません。ここを埋めます。

この記事は[前作（Qiita）](https://qiita.com/axiompro70/items/dc2a839a8f3618d0da12)の姉妹編です。前作は「構文パターンで"握り潰し"の候補は出せるが、正当か欠陥かの最終裁定は人」を実測した話。本作はその一段手前——**候補を出すLLM自身が、実行ごとに違うことを言う**——を測ります。

## 2. 実験設定（統制条件）

> 要点：欠陥を仕込んだ小コードを、構造化出力（JSON）で N 回監査させ、`(行番号, カテゴリ)` の複合キーで"同じ指摘か"を判定する。

- **モデル** = `Qwen2.5-Coder` 1.5B / 7B（Apache-2.0）、`Ollama` で実行。選定理由：無料・ローカル完結・クレカ不要。
- **出力形式** = Ollama の structured outputs（JSONスキーマ強制）。各指摘は `{line_number, category, severity, description}`。カテゴリは列挙型に制約。
- **"同じ指摘"の定義** = `(line_number, category)` の**複合キー**。自然文の言い換えは無視し、「コード上の場所×問題種別」だけで同一性を判定する。埋め込みのコサイン類似度は使わない（意味の近さは"12行目のnullチェック漏れ"と"45行目のnullチェック漏れ"を誤って同一視するため）。
- **対象コード** = 4種：`target_a`（欠陥7件）、`clean`（欠陥なし・偽陽性の基準）、`decoy`（危険に"見える"が文脈上は実害の低い構造——当初の想定。§8④で見直す）、`easy`（誰でも分かる欠陥6件）。
- **グラウンドトゥルース（GT）** = コードの外の `gt.csv` に事前登録（**コード内・プロンプト内には欠陥のヒントを一切書かない**——理由は §8①）。
- **二層再現** = 第1層（生成・非決定・著者が1回だけ）はログを凍結。第2層（集計・決定的）は読者が再現。

## 4. 手法（測り方の要点）

> 要点：信頼性（実行どうしが一致するか）と妥当性（正しいか）を、**配線レベルで分離**する。

- **信頼性 = Jaccard係数（GTを使わない）**。`Jaccard = 一致した指摘数 / 両実行で出た指摘の種類数`。1.0＝完全一致、0＝全く別。**行ズレ許容幅 $w$**（±0＝厳格／±2・±5＝それだけ行がズレても同じ指摘とみなす）を変えて併記。マッチングは**最適二部マッチング**で対称に計算（素朴な貪欲法だと突き合わせる順で値が変わる。これも開発中に実際に踏んで直したバグだ）。同一条件内の**全runペアの平均**を報告する。
- **妥当性 = precision / recall / 偽陽性数（GT基準）**。欠陥なし `clean` で偽陽性を測り、`decoy` では「GTには入れていないが静的解析なら警告し得る灰色スメル」がどれだけ定着するかを見る。recall/precision は特記なき限り **target_a・カテゴリ×行±2でGT一致・空run除外のper-run平均**。
- **多数決** = 「N回中 k回以上で出た指摘だけ採用」。閾値 k と、キーの粒度 $w$ の**2軸**で真陽性・偽陽性の増減を見る。
- **不確実性** = 95%CIは**実行（run）を再標本化**するブートストラップで求める（指摘単位でなく実行単位・種固定＝集計は決定的）。別の注意として、同一コードのN回監査は**コード母集団への一般化を許さない**（擬似反復）＝これらの数値はコードを跨いだ主張ではない。

詳細な指標定義・スクリプトはリポジトリ参照。

## 5. 結果①：固定すれば、決定的だった

> 要点：temp0・seed固定・GPU固定なら、指摘集合は実行間で完全一致した。

温度0の4条件（2モデル × コード `target_a`・`easy` の2種、N=11。決定性の確認が目的なので2コードで足りる）すべてで、$w$±0/±2/±5 の Jaccard が **1.00**（実行間で変動ゼロ、95%CIも退化して [1.0, 1.0]）。さらに **seed を 1・2・3・101・202・303 と変えても出力はバイト単位で一致**しました（この seed 検証は第1層＝`scripts/seed_probe.py`・要 Ollama で実施し、ログは `results/seed_probe.txt` に同梱。§10 の集計が読む `audit_logs.jsonl` とは別の artifact です）。temp0 は貪欲デコード（argmax）なので、理屈の上でも seed は呼ばれません——それを実データで確認できたことになります。（温度0は決定性の確認が目的なので、少数 N=11 で足ります。）

つまり「LLMは常に大きく揺れる」は俗説です。ここは**設定（前提）**であって、発見ではありません。

<details><summary>📐 「完全に決定的」ではない、という留保</summary>

厳密には、別のスパイクで temp0 でも1回だけ1つの指摘のカテゴリが揺れる残差を観測しています。ただしそれは**初回runのみ**で、その run は他より遅延が長く（cold-cache/ウォームアップの兆候）、**ウォームアップを入れた seed 検証では残差ゼロ**でした。機序はおそらくGPU由来（並列リダクション順序等）ですが、cold-cache由来の可能性も切れておらず、**特定はできていません**（いずれにせよ seed ではありません）。参考：[Thinking Machines "Defeating Nondeterminism in LLM Inference"](https://thinkingmachines.ai/blog/defeating-nondeterminism-in-llm-inference/) は推論サーバの非決定性の主因を**バッチサイズの変動**とし、バッチが固定なら再現するとします——本記事のローカル単一リクエスト（バッチほぼ固定）で 1.00 が出たのはこの見立てと整合します。なお先行研究 [Klishevich ら](https://arxiv.org/abs/2502.20747) が「温度0でも応答が揺れた」としたのは**クラウドAPI＝バッチが変動する条件**での話で、本記事のローカル固定バッチとは対照的。矛盾ではなく、同じ機序（バッチ不変性）の予測通りです。

</details>

## 6. 結果②：温度を上げると崩れる（そして"採点表"がプロンプトに漏れていた話）

> 要点：候補を広げる目的で温度を上げる（temp0.7）と、同じコードでも指摘がバラつく。

temp0.7・欠陥入り `target_a` の Jaccard は 7B **0.52** [0.47, 0.59]、1.5B **0.07** [0.04, 0.11]。固定時の 1.00 から大きく崩れます。しかも 7B は 0.52 が**行ズレを許すと 0.80**（±2）まで戻る——**不一致の正体の大半は「同じバグを違う行に指す」ドリフト**です。

低 recall の内訳を「行だけ一致」で測り直すと、崩れ方に2種類あることが見えます。**7B の`target_a`は per-run の"行だけ一致"recall（±2まで許容）≈ カテゴリ厳格recall（0.31≈0.31）**＝±2内でも当たっていない＝見逃しの大半は本物の非検出（カテゴリ違いでもドリフトでもない）。一方 `easy` や 1.5B ではカテゴリ recall が行 recall を大きく下回る＝**毎回ラベル（カテゴリ）が変わる**。複合キーの揺れは「行」だけでなく「カテゴリ」でも起きています。

<details><summary>📐 数値の詳細と、"位置は当てている"を過大評価しないための注意</summary>

| コード | モデル | recall(カテゴリ厳格) | recall(行のみ) |
|---|---|---|---|
| target_a | 7B | 0.31 | 0.31 |
| target_a | 1.5B | 0.05 | 0.18 |
| easy | 7B | 0.42 | 0.81 |

**⚠ union（51回の和集合）で"全欠陥の行を検出"は偶然の産物になりやすい**：1.5B は `target_a` の**32行中25行**を51回のどこかで叩いています。これだけ広く撃てば、7つのGT行が和集合に含まれるのはほぼ自明で、"位置を当てる能力"の証拠にはなりません。だから本文では union のカバレッジを能力の証左に使わず、per-run の recall と precision で語ります。

</details>

**ここで最初の大きな誤りを踏みました。** 当初 `target_a` には、自分用のメモとして `# DEFECT(L9): SQL injection` のようなコメントを欠陥行に書いていました。監査対象のコードをそのままプロンプトに渡す設計なので——**モデルは私の"解答"を読んでいた**わけです。ラベルを消したら recall が崩落し、実行間のバラつきはむしろ深くなりました。詳しくは §8。

## 7. 結果③（核心）：多数決は"正しさ"でなく"しつこさ"を拾う

> 要点：「N回やって多数決」は安全網に見えて、そうではない。**採否を決めているのは"票の集まりやすさ"であって、"正しさ"ではない**。

多数決 = 「N=51 中 26回以上（過半）で出た指摘だけ採用」。7B が51回のうち各指摘を何回出したか（＝票数）を、多数決線と重ねたのが下図です。

![51回の監査で各指摘が何票入ったか（7B・temp0.7）。赤い灰色スメルは多数決線の上に生き残り、青い真バグ ResourceLeak は26/19に割れて線を跨ぐ。](/images/nofreelunch.png)
*図：各指摘の得票（51回中）。青＝GT上の真の欠陥、赤＝GT外の"灰色スメル"（`eval`/`shell=True`）。破線＝多数決の閾値（26）。真バグ ResourceLeak だけが行16/17に割れて線を跨いでいる。*

まず**機序**です。同一性のキーを `(行, カテゴリ)` にすると、**同じバグでも指す行が1行ズレれば"別の指摘"として別々に集計されます**。図の ResourceLeak は、7Bが **45/51 回**（＝ほぼ毎回。行16か17を指した回数）検出しているのに、票が**行16に26・行17に19**へ割れている。行を問わず「ResourceLeak と言えたか」まで緩めると **49/51 回**——ただし指した行は 12・15・16・17・18・21 の**計6通り**に散っています（これがドリフトの実像です）。多数決は"同じ鍵"の票しか数えないので、割れた瞬間、**45回見つけていても各行は26と19**。閾値kを動かすと、この綱渡りが見えます：

| 閾値 k | 行16（26票） | 行17（19票） | 採用結果 |
|---|---|---|---|
| k=19 | 採用 | 採用 | 両方残る（重複気味） |
| **k=26（過半）** | 採用 | 不採用 | ぎりぎり片方だけ残る |
| **k=27** | 不採用 | 不採用 | **45/51回見つけた本物が落ちる** |

多数決の可否が、モデルの**検出能力ではなく"鍵の割れ方と閾値"に支配されている**のが分かります。これは1件の逸話ではなく、7B・target_a で**1回以上検出したGTの約4割が実行間で行ドリフトを示す**（line_drift_rate 0.4＝分母は「1回以上検出されたGT」）という、率で観察された現象の代表例です。なお、**割れない本物は普通に通ります**——`SQLInjection`（行9）は **51/51 の満票**（図の左端の青バー）。多数決が落とすのは「見つける能力が低い本物」ではなく、「**表現が揺れる本物**」です。

一方、**赤の"灰色スメル"は割れずに生き残ります**：`eval("2 + 2")` への警告 33/51、`subprocess.run(..., shell=True)`（定数）への警告 28/51。どちらも定数を渡すだけで実害は低いのに、モデルは"パターン"に反応して一貫して同じ場所を指すからです（余談：後者をモデルは `CommandInjection` でなく `CodeInjection` と分類していました＝カテゴリ自体も揺れる、§6の話とつながります）。

これを**キー粒度 $w$（±0/±2/±5）× 閾値**の面で見ると、トレードオフが立ちます。

| コード | 多数決後（k=26） $w$±0 | ±2 | ±5 | 読み |
|---|---|---|---|---|
| `clean`（欠陥なし）7B | 偽陽性 **0** | 0 | 0 | 低頻度の偽陽性は多数決で消える（狙い通り） |
| `decoy`（灰色）7B | 定着指摘 **2** | 2 | 2 | 一貫した指摘は生き残る（真偽は §8④） |
| `target_a`（欠陥）7B | 真陽性 2 | 2 | 3 ※ | ※+1は別欠陥の取り違え（下記） |
| `easy`（欠陥）7B | 真陽性2/偽陽性1 | 3/2 | 3/1 | キーを緩めると真陽性↑だが偽陽性も一時↑ |

割れた票はキーを緩めれば救えますが、利得はタダではありません：`easy` では偽陽性が一時 1→2 に増え、`target_a` では広すぎる窓が**別個の欠陥（接続リークとファイルリーク）を取り違えて同一視**します（上の 2→3 の"復活"はこの取り違えを含み、真の利得ではない）。派手なトレードオフではありませんが、**"ただ緩めれば無料で救える"わけでもない、という非対称なコスト**があります（no free lunch）。

そして`clean`には、それ自体面白い含意があります。7Bは欠陥なしコードの**37/51回（72.5%）で"指摘なし"**。しかし**何か言うときは毎回違うことを言う**——非空の clean 実行どうしの一致はわずか **0.12** です。"言いがかり"にすら一貫性がなく、だから多数決で消える。裏を返せば、多数決が消せるのは"バラバラな"偽陽性だけで、`decoy` のように**一貫した**灰色指摘は消せない、ということです。

<details><summary>📐 $w$±5 の"復活"が過剰計上である理由</summary>

$w$±5 では ResourceLeak の指摘（多くは行16）が一つの大きなクラスタに束ねられ、その窓が接続リーク（6–11行）とファイルリーク（15–17行）の**両方に重なる**ため、1つのクラスタが2つの別GTに信用計上されます。つまり「真陽性が2→3」の+1は、モデルが実際に接続リークを見つけた証拠ではなく、窓を広げた副作用です（接続リーク領域はどの実行でも一度も当たっていません）。

</details>

**結論：多数決が選別しているのは"真偽"ではなく"しつこさ（パターンの顕著さ）"です。** 一貫して出る指摘は、それが灰色スメルでも本物のバグでも生き残る。逆に、本物でも**指す場所がブレる**指摘は落ちうる。「しつこさ」と「正しさ」は同じものではなく、多数決は前者を強く拾う。

> ただし正直に：この観察は、本標本では主に1件のドリフト真バグ＋2件の灰色スメルで立っています。統計的独立を検定したわけではなく、**一般則ではなく"観察された機序"**として読んでください。そして——**これは多数決そのものの否定ではありません**。問題は、`(行, カテゴリ)` のような揺れやすいキーを**そのまま票として機械的に採否する**ことです。キーの取り方（カテゴリのみ／AST節点／欠陥領域）を変えれば挙動は変わります＝これも「閾値×キー粒度」の no free lunch の一部です。

**実務への落とし込み**：LLMレビューを1回で終わらせず、複数回の出力を「自動採決リスト」ではなく「**候補プール**」として扱う。高頻度の指摘は優先確認し、低頻度でも重大カテゴリなら捨てない。最後に、テスト・静的解析・人間の仕様理解で**裁定**する。

## 8. 落とし穴ログ（測ってから語る・測定器の故障も含めて）

> 要点：この検証で踏んだ5つの誤り。どれも「同じ罠を踏む人」への一次情報。**特に③は"測定器そのもの"のバグ**で、この記事の白眉です。

- **① 解答がプロンプトに漏れていた**。誤=「recall が高い、いいベンチマークだ」→ 正=「モデルは欠陥コメントを読んでいただけ」。欠陥行に書いた `# DEFECT: ...` コメントごと監査対象に渡していた。**削除したら recall は崩落し、バラつきは深まった**。教訓：ベンチの正解を、テスト対象のモデルから見える所に置いてはいけない。
- **② CPUだと思っていたら100% GPUだった**。誤=「ローカルCPU推論。スレッド数（1 vs 4）を変数にしよう」→ 正=「`ollama ps` が 100% GPU（RTX 4060 Ti/CUDA）」。**`num_thread` はCPU推論でしか効かず、100% GPUオフロード環境では無視される**（Ollama利用者が「スレッド数を変えたのに結果も速度も変わらない」と混乱する典型）。私のスレッド軸は何も測っていなかった。教訓：使っているアクセラレータを"推測"せず provenance に記録する（記録した瞬間に間違いに気づいた）。
- **③ 自分の集計器に、直したそばからバグを入れた（測定器の故障）**。クラスタリング補助関数を一行に"最適化"したとき、`リストをappend → 同じ参照をclear` という古典的なミューテーションバグを埋め込み、**多数決の真陽性・偽陽性カウントを静かに壊していました**。気づけたのは、**ある指摘数を手で数えたら 2、ツールは 3、と食い違ったから**。**この泥臭い手動検算が無ければ、壊れた数字で記事を書いていた。** 自動集計を盲信せず、数件を手で数え、全ての主要数値を集計コードとは別に独立再計算（同梱の `verify_independent.py`）して初めて捕まえました。教訓：**"修正"こそ検証が要る**。
- **④ 私の"偽陽性"は、実は本物のスメルだった**。誤=「decoy は無害。だからモデルの指摘は偽陽性」→ 正=「`shell=True`（定数）・`eval`（定数）は Bandit 等が実際に警告する本物のコードスメル」。つまり多数決を生き残った"偽陽性"は、むしろ一貫して正しい指摘だった可能性がある。この矛盾が、逆に核心の主張を強くしました——だから主張を「偽陽性が生き残る」から「**しつこさが（真偽と無関係に）生き残る**」に組み替え、decoy の指摘は "偽陽性" ではなく "GT外の定着指摘" と呼ぶことにしました。
- **⑤ 測った本人の記憶が、結果を捏造した**。誤=原稿の初期版に「`md5` への警告も多数決を生き残った」と書いた → 正=ログでは 7B・decoy で md5 系の票は **51回中0回**（生き残った定着指摘は `eval` 33票・`shell=True` 28票のみ）。④の"生き残った灰色スメル"の印象に引きずられ、ありもしない数字が頭の中でできあがっていた。公開前にログへ突き合わせて発覚・削除。教訓：**測った本人の記憶も一次ソースではない**。数字は、書くたびに生ログへ戻って確認する。

締め：これらは「LLMの出力を鵜呑みにしない」を、**自分の測定器と自分の主張にも適用した**結果です。数値は全て、集計コードとは独立に再計算して一致を確認しています。

## 9. 限界（測っていないものは測ったと書かない）

- **標本が小さい**：N=51、モデル2サイズ、`RTX 4060 Ti / CUDA / Q4_K_M / Ollama 0.31.1` の**単一設定**。CPU実行・他GPU・他量子化では挙動が変わり得る。
- **核心の主張は事例が少ない**：「しつこさ ≠ 正しさ」は主に1件のドリフト真バグ＋2件の灰色スメルで立つ。機序としては筋が通るが、一般則として過信しない。
- **「弱いほど崩れる」は仮説**：1.5B < 7B は2点の比較で、能力と分散が交絡している。傾向であって確定ではない（3サイズ目を足せば"傾向"になる）。
- **recall はカテゴリ厳格で低い**：「場所は当てるがラベルが揺れる」分を分離して開示した。絶対的な監査品質のベンチではない。
- **decoy の"偽陽性"は議論の余地あり**（§8④）。だから主張を"しつこさ"で立て、"無害/有害"の断定は避けた。
- **因果は主張しない**：射程は「無料・ローカル小型モデルの自コーパス実測」まで。
